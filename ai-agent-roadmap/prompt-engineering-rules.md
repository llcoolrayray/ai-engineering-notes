## R01 · 模糊 = 歧义,LLM 会按概率采样不同解读

**习得阶段**:阶段一(2026-05-08)

**类型**:行为认知 — 关于 LLM 是什么、如何工作

**实验证据**:
- Prompt:`翻译:你好`(没指定目标语言)
- DeepSeek V4 Flash 三轮输出:`Hello` / `你好` / `Hello`
- 第 2 轮 LLM 把"你好"返回成"你好"——它合理地解读为"原样返回中文"

**核心教训**:
> LLM 不是"做对/做错",是"**在多种解读里按概率选一个**"。
> 同一个 prompt 多次调用,采样路径可能不同。

**工程含义**:
- 写 prompt 时用**排除法**——主动列举可能的歧义,逐一封堵
- 不是"我写得明白就行",是"**LLM 看到这话会理解出几种意思**"
- 阶段四 Agent 工具调用时,这条会变成"工具描述必须排除歧义"

**反例**(Java 思维):
> "我说'翻译你好',它当然知道翻译成英文"——错。LLM 不知道你的隐含假设。

## R02 · 明确目标语言/格式/范围

**习得阶段**:阶段一(2026-05-08)

**类型**:写法 — 关于怎么写 prompt

**实验证据**:
- Prompt 加上 `把这句话翻译成英文` + `只返回翻译结果,不要加任何引号或解释`
- DeepSeek 输出立刻收敛到 `Hello`(虽然有一次出现 `Hello.` 带句号)

**有效的"明确性"维度**:
1. **目标语言/类型**:中→英、问题→答案、长文→摘要
2. **输出长度**:不要解释、用一句话、最多 3 点
3. **输出格式**:JSON、Markdown 表格、单行文本

**工程含义**:
> 把 prompt 当作 **API 文档** 来写——指定输入输出契约。
> 不要写成"给同事看的描述"。

**和 R04 的关系**:即使你写了约束,LLM 也不一定 100% 遵守(R04)。但有约束 vs 没约束,稳定性差距巨大。

## R03 · few-shot 是最强的格式控制

**习得阶段**:阶段一(2026-05-08)

**类型**:写法 — 当指令难以描述时的杀手锏

**实验证据**:
```
翻译中文为英文。

示例:
输入:谢谢
输出:Thanks

输入:再见
输出:Goodbye

输入:你好
输出:
```
- DeepSeek 严格输出 `Hello`,**对齐示例风格**(无引号、无句号、单词)

**核心洞察**:
> LLM 对"看到什么样,就输出什么样"的**模仿能力** >> 对自然语言指令的**理解能力**。

**工程含义**:
- 别花 1 小时调措辞,直接给 3 个示例
- few-shot 是阶段一最能带走的最强工具
- 阶段二会学 zero-shot vs one-shot vs few-shot 的工程权衡(token 成本 vs 稳定性)

**注意事项**:
- 示例的**多样性**比数量重要——3 个差异化示例 > 10 个雷同示例
- 示例本身不能矛盾,否则 LLM 不知道学哪个

## R04 · LLM 的"约束"不是硬保证

**习得阶段**:阶段一(2026-05-08)

**类型**:行为认知 — Java 工程师必须接受的现实

**实验证据**:
- Prompt 明确说:`不要加任何引号或解释`
- DeepSeek 第 2 轮仍返回 `Hello.`(带句号)
- LLM 没违反"不加引号、不加解释",但加了你没禁的"标点"

**核心教训**:
> 约束是**降低出错概率**,不是**杜绝出错**。
> Java 里 `assertEquals("Hello", x)` 是事实判断,LLM 里 `不要加 X` 是概率偏置。

**Java 工程师惯性陷阱**:
| Java 直觉 | LLM 现实 |
|---|---|
| 写了规则 → 必然遵守 | 写了规则 → 大概率遵守 |
| 不遵守 = bug | 不遵守 = 正常分布的尾部 |

**工程含义**:
- 关键场景**必须做后处理校验**——剥引号、剥空白、剥标点再比较
- 永远不要让 LLM 输出**直接进入业务逻辑**——之间必须有校验/解析层
- 阶段五"可靠性"会专门讲 fallback 设计

**对 CVE 监控 Agent 的具体影响**:LLM 提取产品名时不能假设"它一定按要求格式返回"。

## R05 · JSON 输出要么纯净,要么被 markdown 包

**习得阶段**:阶段一(2026-05-08)

**类型**:输出处理 — 不同模型差异巨大的细节

**实验证据**:
- Prompt:`用 JSON 格式返回,字段为 source 和 translation,不要任何其他文字`
- DeepSeek V4 Flash:`{"source": "你好", "translation": "Hello"}` — 三轮纯 JSON,完全一致 ✅
- Gemini 2.5 Flash:` ```json\n{...}\n``` ` — 被 markdown 代码块包裹 ❌

**核心教训**:
> 同一个 prompt 在不同模型上,序列化稳定性**差异巨大**。
> 靠 prompt 哄 JSON 在生产是赌博。

**工程对策**:
1. **临时方案**:写 robust 的解析函数,先 `strip("```json\n")` 再 `json.loads`
2. **正确方案**:阶段二会学 **JSON Mode / Structured Output API**——厂商提供的"硬模式",直接强制 JSON 输出
3. **生产方案**:用 Pydantic 定义 schema,SDK 自动校验 + 重试

**对应 SDK 能力**(待阶段二验证):
- OpenAI:`response_format={"type": "json_object"}`
- Anthropic:`tools` + `tool_use` 强制结构
- DeepSeek:支持 OpenAI 的 `response_format`(待验证)
- Gemini:`response_mime_type="application/json"` + `response_schema`
