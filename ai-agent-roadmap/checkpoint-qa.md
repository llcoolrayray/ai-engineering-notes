### Q1 · 概念检查 — LLM 无状态的含义

**问题**:
> "LLM API 是无状态的,所谓对话是每次把历史发过去"——请用自己的话解释这意味着什么?如果用户说了 5 句话,客户端代码每次发请求要怎么做?

**回答**:
> 非常理解,HTTP 本身也是无状态的。在 AI 中,无状态指的是它没有记忆,不会记住你上一次和它聊了什么。所以在你和它的对话中,每一轮对话都需要你将当前和之前的所有对话记录都发送给它,才看起来好像 AI 有记忆。

**评价**:✅ **完整且准确**

**Java 工程师视角**:HTTP 无状态的类比抓到了本质。

**需要补充的事实**:
> 阶段四学 Agent 记忆系统时会发现"每次发完整历史"有上限——LLM 有**上下文窗口限制**。
> 比如 DeepSeek V4 是 1M token,听起来很大,但聊几百轮就会撑爆。
> 届时要学"如何裁剪历史",这是阶段四的核心技术之一。

### Q2 · 代码检查 — `response.choices[0].message.content` 设计

**问题**:
> 看 hello_llm.py 里这一行:`response.choices[0].message.content`。
> `[0]` 是什么意思?为什么 `choices` 是数组?不是 `[0]` 行不行?

**回答**:
> choices 是数组,0 表示第一个元素。

**评价**:⚠️ **答对了但太简短**(只答了 1/3 子问题)

**需要补全**:
1. `[0]` 是第一个元素 ✅(答了)
2. **为什么 `choices` 是数组?** OpenAI 接口设计支持参数 `n=3`(让 LLM 同时给 3 个候选答案,做多样性采样)。如果不传 n,默认 n=1,数组就 1 个元素,所以写 `[0]` 永远拿到那个元素。
3. **不写 `[0]` 行不行?** **不行**——`response.choices` 是 `list`,直接访问 `.message` 会报 `AttributeError: 'list' object has no attribute 'message'`。

**反思**:这道题答太快没展开。"看着懂"和"自己能讲清楚"是两件事——后续答题要展开。

### Q4 · 工具检查 — 无脑重试的工程问题

**问题**:
> tenacity 的 `@retry`,如果让它"任何异常都重试 100 次,每次等 1 秒",工程上有什么问题?至少 2 个。

**回答**:
1. 没有对异常进行细分判断。模型内部出错或网络问题重试合理;参数错误、限流不断高频重试是浪费时间并可能被识别为攻击行为
2. 间隔每次 1 秒不合理,应采用指数退避。否则在大并发下,服务器已经负载很高时还在高频重试,实际上类似攻击服务器
3. 重试时间指数退避也要加上随机数,避免很多请求都是同一时间重试

**评价**:✅ ✅ ✅ **三条都对,加分点:讲到了 jitter(惊群效应)**

**加分点详细**:
- 第 3 条提到的"加随机数避免同时重试" = jitter,是分布式系统的**惊群效应**(thundering herd)防御
- 这一点很多 Java 工程师做了 10 年都没意识到,说明在分布式/高并发上的直觉是有的
- 这种直觉对**阶段四 Multi-Agent** 和**阶段五生产工程**会非常有用

**Java 经验直接迁移**:Spring Retry / Resilience4j 的设计思路完全适用。

### Q5 · Java 工程师惯性检查 — LLM vs Java 的认知差异

**问题**:
> 一句话总结:LLM 工程和 Java 工程最大的认知差异是什么?这个差异在 CVE 监控 Agent 上会带来什么具体风险?

**回答**:
> 对不确定性的认知。Java 编程返回结果是 100% 确定的,但 LLM 存在不确定性,我可以给他约束、给他 prompt,但对 AI 这只是建议不是 100% 的红线。所以后面在开发中需要明白 AI 返回的数据是具有不确定性的。

**评价**:✅ **抓到了核心,但缺具体场景**

**抓到的核心**:认知层面"LLM 输出非确定性"——对的。

**缺的具体场景**(具体到 CVE 监控 Agent):

让 LLM "提取 CVE 描述里的产品名,以 JSON 数组返回"——

**LLM 可能的失败方式**:
- 把 `Apache HTTP Server` 写成 `Apache` / `HTTP Server` / `apache-httpd` / `Apache HTTPd Server`(同一产品 N 种写法)
- "幻觉"——把描述里没提到的产品也加进去
- 数组里混入空字符串、null、解释文字

**对应风险**:
- **误报**:产品没受影响却通知一堆人
- **漏报**:真受影响的产品没识别出
- **程序崩溃**:下游代码假设 JSON 合法

**对应解法**(阶段三/四/五会学):
- **Prompt 工程层**:结构化输出 + few-shot 示例
- **工程层**:产品名白名单校验
- **评估层**:回归测试集(多少 CVE 的产品识别正确率)
